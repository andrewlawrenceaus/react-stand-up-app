# PRD: Participant Invite Links

**Product:** Stand-Up Duck
**Author:** Andrew
**Date:** 2026-03-30

---

## Problem

Team members currently share the team lead's email/password credentials to access the app. This gives everyone full admin access (manage teams, participants, run stand-ups). The desired model is:

- **Team lead** — full access; manages teams, participants, runs stand-ups, runs retros
- **Team member** — limited access; participates in retros, updates their own profile photo

No account creation should be required for team members.

---

## Solution: Per-Participant Invite Links

Each team member gets a unique magic link generated by the team lead. Clicking the link signs them in anonymously via Firebase Anonymous Auth and creates a validated participant session scoped to their identity. They see a limited view of the app with only the actions they need.

---

## User Stories

- As a **team lead**, I can generate an invite link for each participant so I can share it with them.
- As a **team lead**, I can revoke a participant's invite link so I can cut off access if needed.
- As a **team lead**, when a stand-up begins I can generate magic links for all participants at once and copy the full set (formatted as a name + link list) so I can paste it into a Teams chat in one action.
- As a **team member**, I can click my invite link and immediately access the retro without creating an account.
- As a **team member**, I am automatically identified in the retro (no "who are you?" picker).
- As a **team member**, I can update my own profile photo.
- As a **team member**, I cannot access team management, participant management, or stand-up controls.

---

## Feature: Bulk Magic Link Generation at Stand-Up Start

When the team lead begins a stand-up, they need a fast way to get every participant into the session. Rather than copying links one-by-one from the Participants page, a **"Share Links"** action lets them generate all tokens in one click and copy a pre-formatted message ready to paste into Teams.

### UX Flow

1. Team lead starts a stand-up (existing flow).
2. A **"Share Links"** button appears in the stand-up view (e.g. in the stand-up header or a pre-start modal).
3. Clicking it:
   - Generates invite tokens for any participant that does not already have one (idempotent — existing tokens are reused).
   - Displays a modal/panel listing every participant with their link.
   - Provides a **"Copy all"** button that writes a plain-text block to the clipboard, e.g.:
     ```
     Stand-up links for today's session:

     Alice — https://app.example.com/join/abc123
     Bob   — https://app.example.com/join/def456
     Carol — https://app.example.com/join/ghi789
     ```
4. Team lead pastes the block into the Teams chat. Participants click their own link.

### Design Decisions

- **Idempotent token generation** — if a participant already has an `inviteToken`, use it; otherwise generate a new one. No tokens are revoked or replaced silently.
- **Copy all format** — plain text (not markdown, not HTML) so it renders cleanly in Teams without formatting artifacts.
- **Participant ordering** — matches the existing participant list order.
- **No auto-send** — the app never sends messages; the team lead always controls paste and send.
- **Origin for links** — use `window.location.origin` at copy-time so the link works in any deployment environment without hardcoding a URL.

### New / Modified Components

| Component | Change |
|-----------|--------|
| `src/components/standup/StandUpHeader.jsx` (or equivalent) | Add "Share Links" button; wire to bulk-generate handler |

## Technical Design

### Auth Model

- Team leads use existing email/password Firebase Auth (unchanged).
- Team members use **Firebase Anonymous Auth** seeded by a token.

### Token Flow

```
Team lead generates token
  → writes participantTokens/{UUID} = {ownerUID, participantId}
  → stores UUID in users/{uid}/participants/{id}/inviteToken

Team member clicks /join/{UUID}
  → app reads participantTokens/{UUID} (public read, no auth required)
  → signInAnonymously() → gets anonymousUID
  → writes participantSessions/{anonymousUID} = {ownerUID, participantId, token}
    (RTDB rules cross-reference validates: token must match ownerUID + participantId)
  → redirects to /retro
```

### Data Structure

New top-level RTDB paths (alongside existing `users/`):

```
participantTokens/{token}
  ownerUID: string
  participantId: string

participantSessions/{anonymousUID}
  ownerUID: string
  participantId: string
  token: string
```

Modified participant object:

```
users/{uid}/participants/{participantId}/
  id, name, photoUrl   (existing)
  inviteToken: string | null   (new)
```

### RTDB Security Rules (`database.rules.json`)

```json
{
  "rules": {
    "users": {
      "$uid": {
        "teams": {
          ".read": "auth != null && auth.uid == $uid && !auth.token.firebase.sign_in_provider.matches('anonymous')",
          ".write": "auth != null && auth.uid == $uid && !auth.token.firebase.sign_in_provider.matches('anonymous')"
        },
        "participants": {
          ".read": "auth != null && auth.uid == $uid && !auth.token.firebase.sign_in_provider.matches('anonymous')",
          ".write": "auth != null && auth.uid == $uid && !auth.token.firebase.sign_in_provider.matches('anonymous')"
        },
        "retros": {
          ".read": "auth != null && (auth.uid == $uid || root.child('participantSessions').child(auth.uid).child('ownerUID').val() == $uid)",
          "$teamName": {
            "active": {
              "items": {
                ".write": "auth != null && (auth.uid == $uid || root.child('participantSessions').child(auth.uid).child('ownerUID').val() == $uid)"
              },
              "$other": {
                ".write": "auth != null && auth.uid == $uid"
              }
            },
            "history": {
              ".write": "auth != null && auth.uid == $uid"
            }
          }
        }
      }
    },
    "participantTokens": {
      ".read": true,
      "$token": {
        ".write": "auth != null && !auth.token.firebase.sign_in_provider.matches('anonymous') && auth.uid == newData.child('ownerUID').val()"
      }
    },
    "participantSessions": {
      "$anonymousUID": {
        ".read": "auth != null && auth.uid == $anonymousUID",
        ".write": "auth != null && auth.uid == $anonymousUID && newData.hasChildren(['ownerUID', 'participantId', 'token']) && root.child('participantTokens').child(newData.child('token').val()).child('ownerUID').val() == newData.child('ownerUID').val() && root.child('participantTokens').child(newData.child('token').val()).child('participantId').val() == newData.child('participantId').val()"
      }
    }
  }
}
```

Key design points:
- Participants can read and write only to `retros/$teamName/active/items` (add items, toggle agree, edit/delete items).
- Participants cannot write categories, timer, history, or complete retros.
- `participantSessions` write rule cross-references `participantTokens` server-side, preventing session forgery.
- `participantTokens` is publicly readable (no auth) so the join page can resolve the token before signing in.

### Storage Rules Change

The current rule requires `request.auth.uid == uid` (path segment). Anonymous users have a different UID. The rule is relaxed to allow any authenticated user to write to `participants/{ownerUID}/{participantId}/photo` — the path structure (containing both ownerUID and participantId as UUIDs) acts as implicit authorization since participants only learn their path from their validated session.

File size (5 MB) and content-type (image/jpeg|png|gif|webp) restrictions remain in place.

## Edge Cases

| Case | Handling |
|------|----------|
| Team lead clicks a join link by mistake | JoinPage detects non-anonymous user → redirects to `/` |
| Invalid or revoked token | `resolveToken` returns null → show error message |
| Participant returns to app later | Anonymous auth persists in localStorage; session reloaded from RTDB |
| Participant signs out ("Leave") | `signOut()` clears session; re-clicking the link creates a new anonymous session |
| Token revoked while participant is active | Existing session continues until they sign out (RTDB validates on write only) |
| Participant tries to navigate to admin pages | Root layout redirects to `/retro` |
